# 第 0 章：Claude Code 到底是怎么工作的（心智模型）

**读完这章你能得到什么：** 一张完整的运行地图。Claude Code 从你按下回车到给出回答，中间经过 7 个环节。这张地图会告诉你每个环节发生了什么、为什么有时候快有时候慢、为什么有些指令会被忽略。

后面每一章的优化技巧，都挂在这张地图的某个节点上——看不懂地图就上路，只会越走越迷。

---

## 0.1 完整工作流：一个永不停歇的循环

Claude Code 的核心是一个叫 queryLoop 的无限循环。不是"你问一句它答一句"的简单问答，是一个 while(true) 的死循环，只要模型觉得任务没完成，就会一直转下去。

完整流程拆成 7 步：

**输入 → 上下文压缩检查 → prompt 组装（7段静态 + N段动态 + 工具定义 + 对话历史）→ API 调用（流式）→ 流式响应边接收边解析 → 工具执行 → 结果回传给模型 → 回到第 1 步继续循环**

一个真实场景帮你理解这个循环。你输入"帮我把 utils.ts 里的 formatDate 函数改成支持时区参数"，模型第一轮会调用 Grep 工具搜索 formatDate 的位置，搜索结果回传后进入第二轮循环，模型调用 Read 工具读取文件内容。

读完之后进入第三轮循环，模型调用 Edit 工具修改代码，修改完成后模型判断任务完成，输出 end_turn 信号，循环结束。3 轮循环，3 次 API 调用，这就是一个简单修改的真实成本。

再看一个复杂场景。你输入"在项目里找到所有未处理的 Promise rejection 并修复"。这种任务可能触发 8-15 轮循环：先 Grep 搜索模式、再逐个 Read 文件确认、然后逐个 Edit 修复、中间可能还会跑 Bash 执行测试验证修复是否正确。

每一轮循环都是一次 API 调用，每次 API 调用都在消耗 token。理解这个循环，你就理解了为什么有些任务费用高——不是模型贪心，是任务本身需要多轮交互。

### 流式执行：不等回复写完就开始干活

Claude Code 用了一个叫 StreamingToolExecutor 的机制。模型的回复是流式传输的——一边往外吐 token，一边解析。只要检测到某个工具调用的参数已经完整了，立刻开始执行，不等整条回复全部生成完毕。

这意味着什么？如果模型一次回复里调了 3 个 Grep 搜索，第 1 个 Grep 可能在第 3 个 Grep 的参数还没吐完的时候就已经执行完并返回结果了。

你在终端里看到的"一边输出文字一边执行工具"，就是这个机制在起作用。

我一开始以为 Claude Code 是等模型完整回复后才执行工具的，结果发现同样的多文件搜索任务，实际执行时间比预期短很多。

看了 StreamingToolExecutor 的工作方式才明白——流式解析 + 提前执行，节省的是等待时间。

### 工具执行的并行规则

只读操作（Read、Grep、Glob 等）可以并行执行，上限 10 个同时跑。写操作（Edit、Write、Bash 执行命令）必须串行，一个执行完才轮下一个。

这条规则直接影响你的使用策略。让模型一次搜索 8 个文件，比让它一个个搜快 8 倍——因为 8 个 Grep 可以同时跑。但让模型修改 8 个文件，速度跟一个个改没区别——因为 8 个 Edit 必须排队。

你在 CLAUDE.md 里写"一次性读取所有相关文件再动手"，模型执行时就会并行读取，节省大量等待时间。这不是玄学，是并行规则决定的。

### 三种停止信号

模型每次回复带一个 stop_reason，它决定循环走向：

| stop_reason | 行为 |
|---|---|
| `tool_use` | 回复里有工具调用，继续循环 |
| `end_turn` | 任务完成，退出循环 |
| `max_tokens` | 输出被截断，自动恢复继续生成，最多重试 3 次 |

max_tokens 这个特别容易踩坑。模型单次输出有 token 上限，如果一次要写很长的代码会被截断。Claude Code 检测到 max_tokens 后会自动拼接上一次的输出，让模型继续生成，但最多重试 3 次——超过 3 次就报错退出。

我遇到过让模型一次性生成 500 行代码的情况，写到一半突然停了，过了几秒又继续——那就是 max_tokens 触发了自动恢复。如果你的任务需要大量连续输出，拆成多个小任务比一次性写完更稳。

## 0.2 System Prompt 的分层结构：静态 7 段 + 动态 N 段

Claude Code 的 system prompt 不是一整坨文本，是精心分层的模块化结构，分为固定不变的 7 段静态内容和根据你的环境动态加载的 N 段动态内容。

### 7 段静态内容（全球共享缓存）

这 7 段内容对全球所有 Claude Code 用户完全一样。Anthropic 把它们做成了全局缓存——只需要计算一次，所有人复用。这是 Claude Code 能快速响应的关键原因之一。

**第 1 段：身份声明。** "You are Claude Code, Anthropic's official CLI for Claude"——确定角色和能力边界。模型从这一段知道自己是一个代码工具，不是通用聊天机器人。

**第 2 段：系统规则。** 输出对用户可见、权限模式如何控制行为、system-reminder 标签的含义。这一段告诉模型"你在什么环境里运行"。

**第 3 段：编码行为准则。** 这段里有一条非常重要的原则——"三行重复代码好过一个过早抽象"。模型不会主动帮你搞设计模式、不会动不动就抽象出 utility 函数。

如果你发现模型写代码偏保守，不是它能力不行，是 system prompt 里就要求它最小复杂度优先。

**第 4 段：执行安全。** 评估每个操作的可逆性。删文件、force push 这种破坏性操作需要用户确认。这段约束模型"什么时候该停下来问你"。

**第 5 段：工具使用指导。** 专用工具优先于 Bash——搜索用 Grep 不用 grep 命令，读文件用 Read 不用 cat。独立的工具调用要并行发出。这段约束模型"怎么用工具"。

**第 6 段：输出风格。** 不用 emoji。简洁。引用代码必须带文件路径和行号。

**第 7 段：输出效率。** 直奔主题，不要开场白，不要客套话。如果你觉得 Claude Code 回复很干脆不客气，就是这段在起作用。

### 动态 section（独立注册、独立缓存）

静态 7 段之后，通过一个 SYSTEM_PROMPT_DYNAMIC_BOUNDARY 分界标记隔开，后面是动态内容。每个 section 根据你的环境独立加载：

- **memory** — 持久化记忆，跨会话保留的知识
- **env_info** — 环境信息（操作系统、shell、工作目录、是否 git 仓库）
- **language** — 语言偏好（决定模型用中文还是英文回复你）
- **output_style** — 输出风格覆盖
- **mcp_instructions** — MCP 服务器的使用说明
- **scratchpad** — 模型的临时草稿本
- **session_guidance** — Agent/Skill 使用指导（何时调用 Agent 工具、Skill 工具）
- **frc** — 大型工具结果处理（Full Result Content，控制长结果的截断和摘要策略）
- **summarize_tool_results** — 工具结果摘要策略（决定哪些工具结果需要被摘要压缩）

每个动态 section 独立注册、独立缓存。你改了语言偏好，不会让环境信息的缓存失效。这套分层缓存机制让 Claude Code 的响应速度比你自己调 API 快，因为大部分 prompt 内容都命中了缓存，不需要重新计算。

## 0.3 CLAUDE.md 的真实位置——99% 的人不知道的事

**CLAUDE.md 不在 system prompt 里。**

它被包裹在 `<system-reminder>` 标签中，作为对话的第一条 user message 注入。从模型视角看，你的 CLAUDE.md 是"用户说的第一句话"，不是"系统级指令"。

为什么不放在 system prompt 里？原因很直接：system prompt 的静态部分是全球共享缓存。全球用户共享同一份缓存，Anthropic 只需要计算一次就能服务所有人。

如果把你的私人配置塞进 system prompt，每个人的 system prompt 都不一样，缓存就废了。这不是技术偷懒，是成本必须——缓存失效意味着每个用户的每次请求都要重新计算完整的 system prompt。

这带来 2 个直接后果。

**后果 1：模型对 CLAUDE.md 的遵循度低于 system prompt。** System prompt 是"铁律"，user message 是"请求"。你在 CLAUDE.md 里写"永远不要用 any 类型"，模型大概率会遵守但不是 100%。而 system prompt 里的规则——比如"专用工具优先于 Bash"——模型几乎不会违反。

**后果 2：你的 CLAUDE.md 不要写跟 system prompt 冲突的指令。** 比如你在 CLAUDE.md 里写"搜索文件时用 grep 命令"——这跟 system prompt 第 5 段"专用工具优先于 Bash"直接冲突。模型会听 system prompt 的，你的指令会被忽略，而且你还不知道为什么。

理解这个位置关系，是写出有效 CLAUDE.md 的前提。第 1 章会详细展开。

## 0.4 这张地图的导航价值

后面每一章都是在这个循环的某个环节做优化：

| 章节 | 对应循环的哪个环节 |
|---|---|
| 第 1 章 CLAUDE.md | prompt 组装阶段——你的配置怎么注入、怎么写才生效 |
| 第 2 章 权限系统 | 工具执行前的权限检查——控制自动化程度 |
| 第 3 章 Token 和 Cache | API 调用阶段——降低每次调用的成本 |
| 第 4 章 Agent 调度 | 整个循环的复制——让多个循环并行工作 |
| 第 5 章 工具系统 | 工具执行阶段——理解模型手里有什么工具、工具说明书怎么影响行为 |
| 第 6 章 长对话管理 | 上下文压缩检查——在有限窗口里塞更多有效信息 |
| 第 7-8 章 扩展系统 | 给循环加新的工具和能力 |


每当你遇到 Claude Code 的"诡异行为"——回复跟你的指令不一致、执行速度突然变慢、工具突然不好使了——回来看这个循环，定位它发生在哪个环节。环节定位对了，翻到对应的章节就能找到解法。
