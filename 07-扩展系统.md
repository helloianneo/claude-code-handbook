# 第 7 章：Skills + Hooks + Memory 扩展系统

Claude Code 出厂自带的能力覆盖 80% 的场景。剩下 20% 靠三套扩展机制：Skills 注入新能力、Hooks 自动化流程、Memory 跨会话记忆。这三套机制组合起来，能把 Claude Code 从"通用工具"变成"你的专属工具"。

---

## Skills

### 7.1 Skill 的本质

Skill 不是工具，不是插件，是 **prompt 注入机制**。

理解这一点很重要。工具是 Claude 调用的外部函数（Read、Bash、Grep），Skill 是一段预写好的 prompt，被注入到对话中后 Claude 按照这个 prompt 行动。

它不新增任何能力，它只是告诉 Claude "接下来你要按这套规则做事"。

触发方式有两种：
- **用户手动触发**：在输入框打 `/skill-name`，比如 `/commit`、`/translate`
- **模型自动触发**：Claude 根据 `when_to_use` 字段判断当前需要某个 Skill，通过 SkillTool 自动调用

两种执行模式：

**inline（默认）**：Skill 内容展开到当前对话，在主 Agent 中执行。好处是能访问当前上下文——你前面讨论的所有内容它都能看到。

坏处是吃主对话的 token 预算。一个写了 500 字 prompt 的 Skill，每次调用都往对话里塞 500 字。

**fork**：在子 Agent 中运行，独立的上下文空间，不影响主对话。适合独立任务——比如代码审查，不需要知道你之前在聊什么，给它代码就够了。

fork 的另一个好处是继承父进程上下文的 prompt cache，成本比 spawn 低。

怎么选？经验法则：**需要前序上下文的用 inline，独立完成的用 fork。**

### 7.2 Skill 文件格式

文件放置位置：
- `~/.claude/skills/my-skill.md` — 用户级，所有项目都能用
- `.claude/skills/my-skill.md` — 项目级，只在该项目生效

文件由两部分组成：frontmatter（YAML 格式的元数据）+ 正文（prompt 内容）。

完整的代码审查 Skill 示例：

```markdown
---
name: code-review
description: 代码审查，检查安全和质量问题
when_to_use: 当用户要求审查代码或做 code review 时
allowed-tools: [Read, Glob, Grep, Bash]
model: inherit
context: inline
---

你现在要做代码审查。检查以下方面：

1. **安全漏洞**
   - SQL 注入
   - XSS
   - 命令注入
   - 敏感信息泄露

2. **代码质量**
   - 错误处理是否覆盖全部边界
   - 命名是否能让三个月后的自己看懂
   - 函数是否超过 50 行

3. **性能**
   - N+1 查询
   - 内存泄漏
   - 不必要的重复计算

输出格式：
## 严重问题（必须修复）
## 改进空间
## 总体评价
```

完整 frontmatter 字段清单：

| 字段 | 类型 | 作用 | 示例 |
|------|------|------|------|
| name | string | Skill 名称，用于 `/name` 触发 | code-review |
| description | string | 简短描述，显示在 skill 列表中 | 代码审查 |
| when_to_use | string | 告诉模型什么场景该自动调用 | 当用户要求审查代码时 |
| allowed-tools | string[] | 这个 Skill 能用哪些工具 | [Read, Glob, Grep] |
| argument-hint | string | 参数提示，显示在 / 菜单 | 文件路径 |
| arguments | object | 结构化参数定义 | 见文档 |
| model | string | 指定模型或 inherit | inherit |
| disable-model-invocation | boolean | 禁止模型自动调用 | false |
| user-invocable | boolean | 是否允许 / 触发 | true |
| hooks | object | Skill 级别的 hooks | 见 Hooks 章节 |
| context | string | inline 或 fork | inline |
| agent | object | Agent 配置 | 见 Agent 章节 |
| effort | string | 推理努力程度 | high |
| shell | string | 指定 shell 环境 | /bin/zsh |
| paths | string[] | 限制文件操作范围 | [src/, tests/] |
| version | string | 版本号 | 1.0.0 |

**关键字段解读：**

`when_to_use` 是最重要的字段。它不是给人看的，是给模型看的。Claude 在每轮对话中会扫描所有 Skill 的 `when_to_use`，判断当前用户的意图是否匹配。

写得太模糊（"通用工具"）模型不知道什么时候该调用；写得太窄（"当用户说'请做 code review 并检查 SQL 注入'"）又会漏掉很多该触发的场景。

`allowed-tools` 是安全边界。一个翻译 Skill 不需要 Bash 权限，给它 [Read, Write] 就够了。权限最小化原则在这里同样适用。

`disable-model-invocation: true` 适合危险操作——比如一个部署 Skill，你不希望模型自作主张触发部署，只允许你手动 `/deploy`。

### 7.2b 更多 Skill 实战示例

**Commit 规范 Skill（`.claude/skills/commit.md`）：**

```markdown
---
name: commit
description: 按 Conventional Commits 规范提交代码
when_to_use: 当用户要求提交代码、commit、或说"提一下"时
allowed-tools: [Bash, Read]
context: inline
---

按以下流程提交代码：

1. 运行 git status 和 git diff --staged 查看变更
2. 如果没有 staged 的文件，先看 git diff 判断哪些该 add
3. 分析变更内容，判断类型：
   - feat: 新功能
   - fix: 修 bug
   - refactor: 重构（不改行为）
   - docs: 文档
   - test: 测试
   - chore: 构建/工具链
4. 生成 commit message：
   - 标题：type(scope): 一句话描述（英文，不超过 50 字符）
   - 空行
   - 正文：为什么做这个变更（中文，不超过 3 行）
5. 展示 message 给用户确认后再执行 git commit
6. 不要 push，让用户自己决定
```

**翻译 Skill（`~/.claude/skills/translate.md`）：**

```markdown
---
name: translate
description: 中英互译，保留技术术语
when_to_use: 当用户要求翻译文本、translate、或说"翻一下"时
allowed-tools: [Read, Write]
context: inline
---

翻译规则：

1. 中文 → 英文，英文 → 中文，自动检测语言
2. 技术术语保留原文并括号注释
   正确：提示缓存（Prompt Cache）
   错误：提示缓存
3. 代码块、变量名、文件路径不翻译
4. 保持原文的段落结构和 Markdown 格式
5. 口语化翻译，不要翻译腔
   正确：这个方法跑起来更快
   错误：该方法具有更优越的执行效率
6. 如果原文有错误，翻译后在末尾标注
```

这两个 Skill 加起来不到 1K tokens，但能省掉你每次 commit 时想格式、每次翻译时重复说明要求的时间。

### 7.3 Skill 来源（4 种）

1. **内置 Skill**：Claude Code 自带的——update-config、keybindings、verify、debug、simplify、remember 等。不需要安装，直接用 `/` 就能看到。

2. **文件系统 Skill**：你自己写的 .md 文件。`~/.claude/skills/` 是用户级，所有项目都能访问。`.claude/skills/` 是项目级，只在该项目生效。

3. **MCP Skill**：MCP 服务器提供的 prompt 类型资源。跟工具不同，这些是预定义的 prompt 模板，通过 MCP 协议分发。

4. **Plugin Skill**：第三方插件提供的 Skill。通过插件系统安装，跟 VS Code 扩展类似的分发模式。

### 7.4 Skill 预算

Skill 列表在 system prompt 中占上下文的 **1%**。

Opus 1M 窗口 → Skill 列表最多占 10K tokens。Sonnet 200K → 最多占 2K tokens。

超出时的降级策略：
1. 截断 description，只保留前几十个字
2. 极端情况下只显示 Skill 名称，不显示描述

实操含义：**description 写 10-20 个字就够了。** 写 200 字的 description 是浪费——模型大概率只看到被截断的前几十个字。

真正影响模型是否自动调用的是 `when_to_use` 字段，description 只是给人看的列表展示。

另一个坑：如果你装了 30 个 Skill，每个 description 平均 100 字，加上 when_to_use 和其他元数据，整个列表可能就到 5K tokens 了。

Sonnet 用户的 2K 预算直接超了。所以不要贪多——**10 个精准的 Skill 胜过 30 个泛泛的 Skill。**

## Hooks

### 7.5 最常用的 4 种 Hook 事件

Claude Code 有 15 种 Hook 事件，但日常用到的就 4 种：

| 事件 | 触发时机 | 你能干什么 |
|------|---------|-----------|
| **PreToolUse** | 工具执行**前** | 自动批准特定工具、拦截危险操作、修改工具参数 |
| **PostToolUse** | 工具执行**后** | 自动格式化刚写的文件、自动跑测试、发通知 |
| **UserPromptSubmit** | 你按 Enter 提交消息时 | 输入验证、自动补充上下文、记录日志 |
| **SessionStart** | 会话启动时 | 初始化环境、加载配置、打招呼 |

完整事件清单（15 种）：PreToolUse / PostToolUse / PostToolUseFailure / UserPromptSubmit / SessionStart / Setup / SubagentStart / PermissionDenied / Notification / PermissionRequest / Elicitation / ElicitationResult / CwdChanged / FileChanged / WorktreeCreate。

但 80% 的 Hook 需求用 PreToolUse 和 PostToolUse 就能覆盖。PreToolUse 管"要不要执行"和"怎么执行"，PostToolUse 管"执行完做什么"。

### 7.6 四种 Hook 命令类型

每个 Hook 可以执行 4 种类型的操作：

**command 类型——执行 Shell 命令：**

最常用。跑 lint、格式化、测试、发通知，都是 command 类型。

特有能力：
- `async: true`——后台运行，不阻塞你继续交互。适合耗时操作比如跑完整测试套件。
- `asyncRewake: true`——后台运行，完成后唤醒模型继续处理。适合"跑完测试后让 Claude 看结果"这种场景。

**prompt 类型——让 LLM 评估：**

不跑命令，而是把当前上下文交给一个小模型（通常是 Haiku）做判断。适合智能审查——比如"这个 Bash 命令是否可能删除重要文件"。

关键优势：用小模型跑，不占主对话的 token 预算，成本极低。

**http 类型——发 POST 请求：**

把事件数据 POST 到指定 URL。适合通知外部服务——Slack 消息、飞书通知、日志收集。

**agent 类型——Agent 验证：**

启动一个独立的 Agent 来做验证。比如每次 FileEdit 后启动一个用 Haiku 模型的 Agent 做代码审查。独立上下文，不影响主对话，成本极低（Haiku 的价格是 Opus 的 1/60）。

所有类型都支持的通用字段：
- `if`：条件过滤，只在匹配时执行。格式是工具名或 `ToolName(参数模式)`
- `timeout`：超时时间（毫秒）
- `statusMessage`：执行时在终端显示的状态信息
- `once`：只执行一次（比如 SessionStart 的初始化）

### 7.7 Hooks 实战示例

**示例 1：自动格式化——文件保存后跑 prettier**

每次 Claude 写入文件后自动格式化，不用你手动跑。

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "if": "FileWrite",
        "command": "npx prettier --write $TOOL_INPUT_FILE_PATH",
        "timeout": 10000,
        "statusMessage": "Formatting with prettier..."
      }
    ]
  }
}
```

逐字段解释：
- `"PostToolUse"`：在工具执行完之后触发
- `"if": "FileWrite"`：只在 FileWrite 工具执行后触发（不是 FileEdit、不是 Read）
- `"command"`：要跑的 Shell 命令。`$TOOL_INPUT_FILE_PATH` 是 Hook 自动注入的变量，指向刚写入的文件路径
- `"timeout": 10000`：10 秒超时。prettier 格式化一个文件通常 1-2 秒，10 秒是安全余量
- `"statusMessage"`：执行时终端底部显示的文字

踩坑经验：一开始我把 `if` 写成了 `"FileEdit"`，结果 Edit 操作也触发 prettier，有时候 prettier 改了格式导致 Edit 的上下文不匹配报错。

后来改成只对 FileWrite 触发——Write 是写全文件，prettier 怎么改都无所谓。

**示例 2：自动运行测试——编辑代码后跑测试**

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "if": "FileEdit",
        "command": "npm test -- --bail 2>&1 | head -50",
        "async": true,
        "timeout": 30000,
        "statusMessage": "Running tests in background..."
      }
    ]
  }
}
```

关键设计：
- `"async": true`：测试在后台跑，你和 Claude 可以继续对话。不加这个的话，每次 Edit 都要等测试跑完才能继续。
- `"--bail"`：遇到第一个失败就停，省时间。不加的话整个测试套件跑完可能要几分钟。
- `"2>&1 | head -50"`：把 stderr 合并到 stdout，只取前 50 行。测试输出可能非常长，全部塞进 Hook 结果会浪费 token。
- `"timeout": 30000`：30 秒超时。如果你的测试套件更大，可以调到 60000。

如果想让 Claude 看到测试结果并自动修复失败的测试，把 `async` 换成 `asyncRewake: true`。

**示例 3：自动批准安全工具——不用每次点 y**

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "if": "Read",
        "command": "echo '{\"decision\": \"approve\"}'",
        "statusMessage": "Auto-approving read operation"
      },
      {
        "if": "Glob",
        "command": "echo '{\"decision\": \"approve\"}'",
        "statusMessage": "Auto-approving glob search"
      },
      {
        "if": "Grep",
        "command": "echo '{\"decision\": \"approve\"}'",
        "statusMessage": "Auto-approving grep search"
      }
    ]
  }
}
```

原理：PreToolUse Hook 的 stdout 输出包含 JSON 格式 `{"decision": "approve"}` 时，权限系统自动放行，不弹确认框。

这个效果等同于在 `alwaysAllowRules` 里加 `["Read", "Glob", "Grep"]`。区别是 Hook 可以加更复杂的条件——比如只自动批准特定目录下的 Read。

```json
{
  "if": "Read",
  "command": "if echo $TOOL_INPUT_FILE_PATH | grep -q '/src/'; then echo '{\"decision\": \"approve\"}'; fi",
  "statusMessage": "Checking read path..."
}
```

这个 Hook 只自动批准 `/src/` 目录下的读取操作。其他目录的 Read 仍然需要你手动确认。

## Memory

### 7.8 Memory 系统概述

Claude Code 有自动记忆功能（auto memory），跨会话保存你的偏好和项目上下文。不需要手动配置，默认开启。

存储位置：`~/.claude/projects/<项目路径>/memory/`

这个路径下有一个 `MEMORY.md` 索引文件，每条记忆一行链接，指向具体的记忆文件（独立的 .md 文件）。索引上限 200 行 / 25KB。

**记忆在哪里生效？** 每次对话开始时，MEMORY.md 的内容被注入到 system prompt 的 `memory` 动态 section 中。Claude 在第一轮对话时就能看到所有记忆内容。

这意味着记忆和 system prompt 一起享受 prompt cache——不增加额外的 API 成本。

但也意味着记忆越多，占用的固定 token 预算越大。25KB 的索引文件大约 6K-8K tokens，这是每次对话的固定开销。

### 7.9 四种记忆类型

| 类型 | 存什么 | 什么时候自动保存 | 示例 |
|------|--------|----------------|------|
| **user** | 你是谁、在做什么、喜欢什么 | Claude 了解到你的身份信息时 | "Ian 是独立开发者，做 AI 产品" |
| **feedback** | 工作方式纠正——你的确认和否定 | 你说"不要这样做"或"对就这样"时 | "产出必须用中文" "不要用 useEffect 做数据获取" |
| **project** | 项目动态、决策、截止日期 | 了解到项目信息时 | "付费指南 4 月 5 日前交付" |
| **reference** | 外部系统的位置和指针 | 了解到外部资源位置时 | "设计稿在 Figma: xxx" |

其中 **feedback 类型是最有价值的**。你每次纠正 Claude 的行为（"不要自动 push""commit message 用英文""文件名不要用中文"），它都会保存为 feedback 记忆。

下次新开会话，这些纠正自动生效。

project 类型有个贴心的细节：如果你说"这个功能下周二前要完成"，Claude 会把"下周二"转成绝对日期（比如 2026-04-07）再存储。

避免两周后看记忆还是"下周二"不知道是哪个周二。

### 7.10 什么不该存

Memory 不是笔记本。以下 5 类信息不该存：

**代码模式和架构。** "项目用 Next.js + Prisma"——读 package.json 就知道了。存了反而会过时，三个月后换了框架，记忆里还写着旧的。

**Git 历史和变更记录。** "上次重构了 auth 模块"——`git log` 随时能查到更准确的信息。

**调试方案和修复记录。** "XX bug 的原因是 YY"——fix 已经在代码里了，测试用例里也有。存在记忆里等于重复记录，而且代码会变但记忆不会自动更新。

**CLAUDE.md 里已有的内容。** 两个地方写同样的规则，如果改了一个忘了另一个，Claude 会收到矛盾指令。

CLAUDE.md 优先级更高，记忆里的同名规则可能被忽略。

**临时任务状态。** "当前做到第 3 步了"——用 TodoWrite 跟踪。TodoWrite 的状态跟着当前会话走，Memory 的内容会带到所有未来会话。

把临时状态存进 Memory 等于给所有未来会话加噪音。

**该存的是什么？** 不会变的偏好和约束：
- 你的沟通偏好（用中文、口语化、不要 AI 腔）
- 硬性工作规则（不要自动 push、commit 前先 review）
- 项目的长期约束（必须兼容 iOS 15+、不能引入新依赖）

### 7.11 管理记忆

4 种操作方式：

**让它记住。** 直接说"记住我喜欢用 conventional commits"或"记住产出必须用中文"。它会自动创建记忆文件并更新 MEMORY.md 索引。不需要指定类型，Claude 会自动分类。

**让它忘记。** 说"忘掉关于 XX 的记忆"。它会删除对应的记忆文件和索引条目。适合清理过时信息——比如你换了技术栈，旧的框架偏好该删了。

**查看记忆。** 直接读文件 `~/.claude/projects/<项目路径>/memory/MEMORY.md`。每行一条，格式是 `[文件名](文件名) — 一句话描述`。点击链接能看到详情。

**关闭记忆。** 在 settings.json 里设 `autoMemoryEnabled: false`。关掉后 Claude 不再自动保存任何记忆。

已有的记忆文件不会被删除，但也不会被读取——相当于全部失效。

还可以用 `autoMemoryDirectory` 改存储路径。比如你想让所有项目共用一套记忆，可以指向同一个目录。

但一般不需要——项目隔离的记忆更干净。
