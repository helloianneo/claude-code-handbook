# 第 2 章：权限系统完全攻略

**读完这章你能得到什么：** 彻底搞懂 Claude Code 的 5 种权限模式各自适合什么场景，知道 auto 模式背后的 AI 分类器怎么运作，学会用 Hooks 配置自动批准来消灭 80% 的弹框干扰。

以及在出问题时能按照权限判断的 4 步流程快速定位原因。

---

## 2.1 五种权限模式深度对比

Claude Code 内置 5 种权限模式。不是"选最宽松的就好"，而是不同阶段用不同模式。下面逐个拆。

### default 模式

日常使用的起点。读操作自动放行，写文件和跑命令都需要你确认。

优点：安全，每一步都在你掌控中。缺点：写 10 个文件要按 10 次确认，节奏碎得像在用对话框编程。

适合场景：刚接手一个不熟悉的项目，或者你在做敏感操作（比如改生产配置），需要每一步都看一眼。


### acceptEdits 模式

比 default 松一档：读操作自动放行，**文件编辑也自动放行**，只有 Bash 写入命令还需要确认。

这个模式的核心逻辑是：文件修改大概率安全（你随时可以 git diff 看改了什么），但 shell 命令的破坏力不可逆（比如 rm -rf）。所以文件放行，命令拦住。

实测体验：开启后确认弹框减少大约 80%，因为大多数操作就是改文件。偶尔跑个 npm install 或 git commit 时弹一下框，完全可以接受。

适合场景：你对当前项目已经熟悉，信任 Claude 的代码修改，只想在执行 shell 命令时做最后一道把关。

**踩坑经历：** 有一次我在 acceptEdits 模式下让 Claude 重构一个模块，它连改了 15 个文件，全部自动放行。改完我一看 git diff，发现它把一个公共组件的接口签名改了，下游 8 个文件全报错。

教训是：acceptEdits 省了确认时间，但你得养成阶段性 git diff 的习惯。每完成一个小目标就 diff 一次，别攒到最后。

### plan 模式

只读只规划，禁止一切写操作。Claude 可以读文件、搜索代码、分析结构，但不能修改任何东西。

很多人不知道这个模式的正确用法。它不是"限制 Claude"，而是"保护你的代码库"。

当你想让 Claude 先看一遍项目、理解架构、给出方案，但不想它在探索过程中顺手改东西——切 plan。

适合场景：代码审查、架构探索、让 Claude 出方案但你手动执行。

我经常在接手新项目的第一个小时用 plan 模式，让 Claude 把项目结构、依赖关系、核心逻辑全摸一遍，输出一份分析报告。

### bypassPermissions 模式

全部自动放行，没有任何确认。看起来很爽，但有硬性前提——必须在沙箱环境中运行，且没有网络访问。

Claude Code 会检测运行环境。你在本地终端选了 bypassPermissions，它会直接拒绝生效。

必须在 Docker 容器、CI/CD 流水线、或其他隔离的沙箱里才行。

适合场景：CI/CD 自动化、Docker 容器内的批量操作、测试环境的自动化脚本。在这些场景里没有人坐在屏幕前点确认，而且环境本身就是隔离的，不怕搞坏什么。

**注意：** 有人试过在本地强行用 bypassPermissions，发现没效果还纳闷。不是 bug，是设计。安全检查不可能被一个模式开关就绕过去。

### auto 模式

最有意思的一种。每个写操作都由一个 AI 分类器实时判断：这个操作安全吗？安全就自动放行，不安全就拦住问你。

体验介于 acceptEdits 和 bypassPermissions 之间——大部分时候你感觉不到它在拦东西。

但真碰到高风险操作（比如删除大量文件、修改系统配置）它会跳出来。

优点：最接近"心流状态"的模式，你专注思考，Claude 专注执行，只有真正危险的时候才打断你。

缺点：分类器有误判的可能，偶尔会把安全操作当危险操作拦住，也可能（虽然极少）放过一个该拦的操作。

适合场景：你在一个熟悉的项目里做日常开发，不想被频繁打断，但也不想完全放弃安全网。

### 五种模式速查表

| 模式 | 只读操作 | 文件编辑 | Bash 写入 | 使用门槛 |
|------|---------|---------|----------|---------|
| default | 自动放行 | 需确认 | 需确认 | 无 |
| acceptEdits | 自动放行 | 自动放行 | 需确认 | 无 |
| plan | 自动放行 | 禁止 | 禁止 | 无 |
| bypassPermissions | 全部自动 | 全部自动 | 全部自动 | 必须沙箱+无网络 |
| auto | 自动放行 | AI 分类器判断 | AI 分类器判断 | 无 |

## 2.2 Shift+Tab 切换和最佳实践

在输入框里按 Shift+Tab，模式会按这个顺序循环切换：

```
default → acceptEdits → plan → bypassPermissions → auto → default
```

不需要背。按 Shift+Tab 后看底部状态栏，显示的就是当前模式，到你想要的就停。

**什么时候切什么模式，我的习惯是：**

- 刚进入一个项目 → plan，先让 Claude 读一遍代码
- 开始写代码 → acceptEdits，省掉文件确认的干扰
- 进入心流状态、任务明确 → auto，让 AI 分类器兜底
- 做敏感操作（改配置、删文件）→ 切回 default，每步确认
- CI/CD 环境 → bypassPermissions，全自动

一个小细节：Anthropic 内部员工的循环顺序不一样，他们跳过了 acceptEdits 和 plan，直接在 default、bypassPermissions、auto 之间切。你用外部版本不受影响，知道就行。

## 2.3 auto 模式深度解析

auto 模式背后不是一堆 if-else 规则，而是一个实时运行的 AI 分类器（叫 Transcript Classifier）。这个区别很重要，因为它意味着判断逻辑是动态的，会根据上下文变化。

### 分类器怎么工作

每一次写操作（编辑文件、执行命令）触发时，分类器会拿到：当前对话上下文 + 这次具体要做什么操作 + 操作的参数。然后它实时判断：allow 还是 deny。

allow → 操作自动执行，你完全无感。
deny → 操作被拦截，弹框问你要不要继续。

### 连续拒绝追踪机制（Denial Tracking）

这是 auto 模式最精妙的设计。分类器不只是单次判断，它还维护了一个连续拒绝计数器。

每次分类器判定 deny，计数器加一。如果连续拒绝次数超过阈值，auto 模式会整体回退到手动确认——也就是说，后面的操作不再过分类器了，全部弹框。具体阈值未公开，但根据研究推测在 3-5 次左右。

为什么要这样设计？因为连续被拒绝说明你当前的操作模式整体有风险。可能你在批量删文件、可能你在改系统级配置、也可能是你的 prompt 里有分类器不放心的措辞。

与其一个一个拦，不如直接切到全部手动，更安全。

### 实际场景

你在做正常开发，auto 模式全程静默放行。然后你说"把这个目录下所有测试文件删掉"，分类器开始连续 deny——删第一个文件 deny，删第二个 deny，到第三个的时候直接回退到手动模式，后面每个操作都弹框。

这不是 bug。这是分类器在说："你的操作模式变了，我不确定后面还会做什么，我选择把控制权交回给你。"

### 怎么恢复

两个办法。第一，把大批量操作拆成小步骤。一次删 50 个文件会触发回退，分 5 批每批删 10 个，中间做其他操作让计数器重置，分类器更容易放行。

第二，如果你确信操作安全，临时切到 acceptEdits 或 bypassPermissions（沙箱内），做完再切回 auto。

**踩坑经历：** 有一次我让 Claude 在 auto 模式下重命名一个模块的所有文件（大概 20 多个），到第 5 个的时候 auto 开始弹框。我当时还以为是 bug，反复重启。

后来搞明白了：连续的文件删除+创建操作触发了 Denial Tracking。解决办法是用 git mv 批量重命名（一条命令搞定），分类器只需要判断一次，顺利放行。

## 2.4 权限判断完整流程

每个工具调用进来后，要过 4 步检查。任何一步拦截，操作就不会执行。

**第 1 步：validateInput()** — 输入合法性检查。参数类型对不对、必填字段有没有缺、格式是否合规。这一步和权限无关，纯粹是数据校验。

**第 2 步：filterToolsByDenyRules()** — 工具级黑名单过滤。你在 settings.json 里配了 deny 规则的工具，在这一步直接被拦掉，后面的检查根本不会跑到。这是优先级最高的拦截层。

**第 3 步：tool.checkPermissions()** — 每个工具自带的权限逻辑。比如 Bash 工具会检查命令是不是只读的（ls、cat 这类），如果是只读就直接返回 allow，不用走后面的通用检查。不同工具的内置逻辑不一样。

**第 4 步：hasPermissionsToUseTool()** — 通用权限系统。这一步最复杂，它会同时考虑 4 个维度：本地 UI 的模式设置、Bridge 远程连接的权限、Channel 通道权限、以及 Hooks + AI 分类器的判断结果。四路竞赛，哪个先返回就用哪个的结果。


理解这个流程的实际价值：当你的操作被拒绝时，你能判断是哪一步拦的。如果是参数格式问题（第 1 步），改参数就行。如果是 deny 规则（第 2 步），去 settings.json 检查配置。

如果是工具内置逻辑（第 3 步），可能需要换一种方式实现。如果是通用权限（第 4 步），切模式或加 Hook。

## 2.5 各工具权限分类速查

不同工具走不同的权限路径，搞混了会浪费时间排查。

**只读工具（任何模式自动放行）：** Read、Glob、Grep、TaskGet、TaskList。这些工具标记了 isReadOnly: true，不管你在什么权限模式下，调用它们都不会弹确认框。

**需要批准的工具：** Bash（写入类命令）、Edit、Write、NotebookEdit。这些走通用权限系统，由你的模式设置决定是自动放行还是弹框。

**特殊检查工具：** Bash 里执行 sed 命令时走一套独立的白名单验证。只有简单的行打印（`sed -n 'Np'`）和基础替换（`sed 's/a/b/g'`）能通过，复杂的 sed 表达式会被拦截。

这个设计是为了防止 sed 的正则能力被滥用来做危险操作。

**始终允许的工具：** TodoWrite、Task 系列、EnterPlanMode。这些工具的 checkPermissions 直接返回 allow，不走通用权限，因为它们本质上不修改你的代码。

**域名级权限工具：** WebFetch 按 `domain:hostname` 规则逐域审批。你批准了 api.github.com，下次访问同一域名自动放行，但访问新域名要重新确认。

**MCP 工具：** 走通用权限系统，和内置工具同等对待。命名格式是 `mcp__<server>__<tool>`，你可以针对特定 MCP 工具配置 deny 规则。

## 2.6 用 Hooks 实现自动批准

Hooks 是权限系统的合法扩展口。在 settings.json 里配一个 PreToolUse hook，就能对特定操作自动放行，不再弹确认框。

### 自动批准所有 git 命令

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "if": "Bash(git *)",
        "command": "echo '{\"decision\": \"approve\"}'",
        "statusMessage": "Auto-approving git commands"
      }
    ]
  }
}
```

`if` 条件的语法是 `ToolName(pattern)`。`Bash(git *)` 意思是：当 Bash 工具被调用，且命令参数匹配 `git *`（以 git 开头的所有命令）时，执行这个 hook。

hook 的 `command` 输出 `{"decision": "approve"}` 就是自动批准，输出 `{"decision": "deny"}` 就是自动拒绝，输出空内容或其他东西就跳过这个 hook、走正常权限流程。

### 自动批准测试命令

```json
{
  "if": "Bash(npm test*)",
  "command": "echo '{\"decision\": \"approve\"}'",
  "statusMessage": "Auto-approving test runs"
}
```

### 自动批准特定目录的编辑

```json
{
  "if": "Edit(src/components/*)",
  "command": "echo '{\"decision\": \"approve\"}'",
  "statusMessage": "Auto-approving component edits"
}
```

这条规则只对 `src/components/` 目录下的文件编辑自动放行，其他目录的编辑仍然走正常权限。

### 多条规则组合

实际使用中你会同时配多条规则。我的日常配置长这样：

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "if": "Bash(git *)",
        "command": "echo '{\"decision\": \"approve\"}'",
        "statusMessage": "Auto-approving git"
      },
      {
        "if": "Bash(npm test*)",
        "command": "echo '{\"decision\": \"approve\"}'",
        "statusMessage": "Auto-approving tests"
      },
      {
        "if": "Bash(npm run lint*)",
        "command": "echo '{\"decision\": \"approve\"}'",
        "statusMessage": "Auto-approving lint"
      },
      {
        "if": "Edit(src/*)",
        "command": "echo '{\"decision\": \"approve\"}'",
        "statusMessage": "Auto-approving src edits"
      }
    ]
  }
}
```

这样一来，git 操作、测试、lint、src 目录编辑全部自动批准。配合 auto 模式使用，基本上你只有在做非常规操作时才会被打断。

**进阶组合策略：** auto 模式 + Hooks 是性价比最高的配法。auto 模式的分类器处理大部分判断，Hooks 覆盖分类器可能误判的常见安全操作。

两层叠加后，正常开发流程中的弹框频率趋近于零，而真正危险的操作（比如删除非 src 目录的文件、执行未知的 shell 命令）仍然会被拦住。
